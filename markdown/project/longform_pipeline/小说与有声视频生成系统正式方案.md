# 小说与有声视频生成系统正式方案

更新时间：2026-05-17

## 一、文档目的

本文档用于明确本项目的产品目标、设计原则、核心流程、视频约束体系、重生成机制和工程实现方向。

本文档的基本判断是：

- 小说生成本身相对轻。
- 视频生成约束明显更重。
- 真正决定系统产品价值的，主要是视频侧的可控性、一致性、可修改性和可重生成能力。

因此，本文档将采用“小说轻约束、视频重约束”的设计思路。

## 二、项目目标

本项目的目标是构建一套可持续使用的生成系统，使用户和团队成员能够：

- 生成符合需求的小说内容。
- 基于小说内容生成有声视频或有声短片。
- 在生成后对结果提出意见，并推动系统按意见进行局部或整体重生成。
- 在流程复杂、细节繁多的前提下，仍然保持产品使用上的简洁性。

本项目追求的不是“模型一次生成一个结果”，而是“系统持续逼近用户想要的结果”。

## 三、总体设计判断

### 1. 小说和视频不是同一难度问题

小说主要是文字生成问题，虽然也需要题材、文风、人物、节奏等控制，但本质上仍然是相对单一媒介的内容生成。

视频不同。视频天然包含更多维度：

- 人物外观。
- 人物年龄感。
- 服装一致性。
- 场景空间关系。
- 镜头景别。
- 机位和运镜。
- 光影与色彩。
- 情绪氛围。
- 配音。
- 字幕。
- 配乐。
- 音效。
- 镜头衔接。
- 视频片段合成。

因此，本项目不应把小说和视频设计成同等复杂度。更合理的方式是：

- 小说侧保留必要约束。
- 视频侧投入主要设计精力。
- 工程能力优先服务视频链路。

### 2. 产品重点不在“能生成”，而在“能改到满意”

用户真正的需求通常不是“随便给我一个结果”，而是：

- 给我一个方向大体正确的结果。
- 我指出哪里不对。
- 系统继续改。
- 最终越来越接近我想要的东西。

所以本系统的核心竞争力应该来自：

- 约束体系。
- 锁定体系。
- 反馈体系。
- 重生成体系。
- 任务和版本管理体系。

### 3. 先做通用能力，不做过重特化

系统需要支持用户提出强参考方向，例如希望在风格、情绪、镜头语言、气质上高度接近某个明确作品方向。

但系统不应起步就为每一类作品做专用分支，也不应起步就把所有低频需求做成重型配置系统。

正确策略是：

- 先抽象共性的约束能力。
- 先服务高频用户需求。
- 当某类需求反复出现时，再沉淀为模板或增强能力。

## 四、产品定位

本项目是一个“创作约束驱动”的一体化生成系统。

它不是单一的写作工具，也不是单一的视频工具，而是一套从需求到内容、从内容到镜头、从镜头到音视频成片、再从反馈回到重生成的完整工作流系统。

主链路如下：

```text
用户需求
-> 项目设定
-> 小说概要
-> 小说正文
-> 正文定稿
-> 分镜生成
-> 视频约束补全
-> 自动对白与音频生成
-> 视频片段生成
-> 合成最终视频
-> 用户反馈
-> 局部或整体重生成
```

## 五、设计原则

### 1. 简单优先

- 默认流程必须短。
- 默认入口必须少。
- 普通用户先走一键主路径。
- 高级控制项只在需要时展开。

### 2. 视频优先

- 小说生成保证可用即可，不做不必要的重设计。
- 主要设计资源投入视频侧。
- 角色、场景、镜头、音频、合成和重试能力优先建设。

### 3. 约束优先

- 没有约束，模型输出就不可控。
- 视频越复杂，越不能依赖一句大提示词。
- 必须把风格、角色、场景、镜头、声音、禁止项拆成结构化约束。

### 4. 用户反馈优先

- 用户不满意是常态，不是异常。
- 系统必须允许用户对每个关键结果提意见。
- 这些意见必须真正进入下一轮生成，而不是只做文本备注。

### 5. 锁定优先

- 用户已经确认的东西必须可锁定。
- 锁定后不能被后续任务轻易覆盖。
- 这是保证生成稳定性的核心条件。

当前实现上，这条原则已经开始落地到真实代码路径：

- 已锁定角色三视图不会被普通预检重生成覆盖。
- 已锁定对白脚本不会被普通对白脚本生成覆盖。
- 已锁定对白/旁白音频不会被普通配音生成覆盖。
- 已锁定镜头首帧不会被普通首帧生成覆盖。

### 6. 工程服务产品

- 异步任务不是为了复杂化，而是为了让长链路可执行。
- 版本管理不是为了完美，而是为了可回退和可比对。
- 可观察性不是为了开发方便，而是为了让用户看得懂系统在做什么。

## 六、用户需求模型

用户通常会有以下类型的需求：

- 我想写一个什么故事。
- 我想让它呈现出什么气质。
- 我希望人物是什么感觉。
- 我希望视频长什么样。
- 我希望声音是什么风格。
- 我不满意某一段、某个镜头或某个角色时，希望只改那一部分。

因此系统应支持三类输入：

### 1. 基础输入

- 作品名称。
- 类型与题材。
- 内容简介。
- 目标长度。

### 2. 风格输入

- 文风方向。
- 视觉方向。
- 情绪方向。
- 参考作品或参考风格。
- 禁止项。

### 3. 修订输入

- 哪里不满意。
- 希望保留什么。
- 希望加强什么。
- 希望删除什么。

## 七、小说侧设计

小说不是本项目最复杂的部分，因此设计原则应是“够用、可修订、不过重”。

### 1. 小说侧目标

小说侧只需要稳定支持：

- 概要生成。
- 用户反馈后修订概要。
- 锁定概要。
- 根据锁定概要生成正文。
- 对正文进行重写。
- 将章节定稿，作为后续分镜和视频的输入。

### 2. 小说侧约束

建议保留以下核心约束即可：

- 题材。
- 世界观。
- 角色。
- 角色关系。
- 情节主线。
- 章节节奏。
- 文风方向。
- 禁止项。

不必在起步阶段把小说侧做成过重的复杂规划系统。小说侧重点是：

- 给视频提供稳定的剧情输入。
- 给分镜提供清晰的场景和事件基础。
- 让用户能快速改正文，而不是陷入过多结构化编辑。

### 3. 小说侧应支持的修改能力

- 整体重写章节。
- 根据反馈修改章节。
- 后续逐步支持局部段落重写。

小说侧不应过度工程化，但必须具备可修改性。

## 八、视频侧设计总则

视频是本项目的核心难点，因此必须单独建立完整约束体系。

视频生成的基本问题不是“有没有模型”，而是：

- 结果是否稳定。
- 结果是否一致。
- 结果是否接近用户要求。
- 结果是否能被局部修改。
- 失败后是否能继续。

因此视频侧必须重点解决六件事：

1. 风格约束。
2. 人物一致性。
3. 场景一致性。
4. 镜头可控性。
5. 音频包装。
6. 局部重生成。

## 九、视频约束体系

### 1. 项目级风格约束

项目级风格约束是所有视频生成的上游控制源。

它至少应包含：

- 画面媒介。
- 总体气质。
- 色彩倾向。
- 光影倾向。
- 镜头语言倾向。
- 情绪方向。
- 参考风格。
- 禁止项。

例如用户提出强参考方向时，系统不应只保存一句“像某作品”，而应拆成结构化风格块，供后续所有生成流程继承。

### 2. 角色约束

角色约束应至少包含：

- 角色名称。
- 年龄感。
- 外观描述。
- 发型与服装。
- 气质。
- 情绪基调。
- 角色关系。
- 声音方向。

视频侧必须尽量保证同一角色在不同镜头中的一致性。

### 3. 场景约束

场景约束应至少包含：

- 场景名称。
- 空间类型。
- 时间环境。
- 天气环境。
- 光照方向。
- 色彩基调。
- 关键道具。
- 空间关系说明。

场景约束的目标不是“生成一张好看的图”，而是让多镜头下的地点保持相对稳定。

### 4. 镜头约束

镜头约束应至少包含：

- 景别。
- 机位角度。
- 运镜方式。
- 主体动作。
- 主体情绪。
- 构图倾向。
- 光照要求。
- 禁止项。

镜头约束不能只停留在自然语言 prompt，应该逐步结构化。

当前实现已经开始从“只靠 `visual_prompt`”往“多源约束叠加”演进，真实进入后端 prompt 的输入包括：

- 项目级视觉风格锁定。
- `character_refs`。
- `scene_refs`。
- 已确认的 `shot_first_frame`。

### 5. 音频约束

音频约束应至少包含：

- 对白内容。
- 说话角色。
- 情绪。
- 声音风格。
- 旁白策略。
- 字幕文本。
- 配乐提示。
- 音效提示。

### 6. 禁止项约束

禁止项应贯穿整个视频链路，而不是只在最后一环补一句。

视频侧常见禁止项包括：

- 不要真人感。
- 不要照片级写实。
- 不要角色变脸。
- 不要服装漂移。
- 不要错误手部。
- 不要多余文字。
- 不要水印。
- 不要标志。

## 十、强参考方向的处理原则

用户可能会提出很强的参考要求，例如希望整体效果尽可能接近某一明确作品方向。

这类需求在产品上必须被支持，因为这是非常真实的创作诉求。

但实现原则应是：

- 不在代码中写死某个作品专用逻辑。
- 不为每个作品预先做一套完整模板系统。
- 先把参考对象转化为结构化约束。
- 允许用户在生成前确认这些约束。
- 允许用户在生成后根据差异继续修正这些约束。

也就是说，系统要支持“高度贴近某种参考方向”，但产品实现上应保持通用和可扩展。

## 十一、视频生成主流程

建议视频主流程采用以下结构：

```text
定稿章节
-> 生成分镜
-> 生成或修订自动对白脚本
-> 准备角色视觉素材
-> 准备场景素材
-> 生成镜头画面或首帧
-> 生成视频片段
-> 生成对白音频
-> 生成字幕
-> 匹配配乐和音效
-> 合成镜头片段
-> 合成 final.mp4
-> 用户反馈
-> 单镜头或局部重生成
```

这里最关键的是：

- 不是每个项目都必须把所有步骤走满。
- 默认应先走最短闭环。
- 需要更高可控性时，再逐步增加角色素材、场景素材、首帧确认和更精细的音频包装。

当前实现状态已经进入以下形态：

```text
定稿章节
-> 分镜
-> 自动对白脚本
-> 角色对白音频
-> 角色三视图
-> 可选镜头首帧
-> 视频任务
-> 本地 worker
-> 即梦文生视频 / 即梦图生视频-首帧
-> 对白、字幕、FFmpeg 合成
-> final.mp4
```

## 十二、产品交互设计

### 1. 主界面原则

主界面应尽量保持单条主流程，而不是拆成太多难以理解的模块。

建议用户看到的主区域为：

- 项目设定。
- 小说生成。
- 分镜生成。
- 音频生成。
- 视频生成。
- 结果预览。
- 反馈与重生成。

### 2. 默认操作方式

默认应支持一条短路径：

```text
填写项目
-> 生成概要
-> 生成正文
-> 定稿章节
-> 生成分镜
-> 一键准备并开始视频生产
-> 预览结果
-> 按意见重生成
```

### 3. 高级控制项

高级控制项不应消失，但应隐藏在按需展开层中，例如：

- 角色视觉锁定。
- 场景参考绑定。
- 镜头结构化编辑。
- 声音角色映射。
- 字幕样式控制。
- 配乐和音效控制。

## 十三、重生成机制

重生成机制是本系统最重要的产品能力之一。

### 1. 重生成层级

系统至少应支持以下层级的重生成：

- 小说概要重生成。
- 单章正文重写。
- 分镜重生成。
- 单镜头重生成。
- 自动对白脚本重生成。
- 单条对白音频重生成。
- 单镜头视频片段重生成。
- 从失败点继续合成最终视频。

当前实现里，已经开始具备“重生成但不乱覆盖”的基础机制，重点不再只是能不能重生成，而是：

- 重生成前是否能识别已锁定内容。
- 预检时是否能跳过已锁定内容。
- 前端是否能让用户明确知道这次跳过了什么。

### 2. 重生成输入

每次重生成都应明确记录：

- 用户不满意的点。
- 本轮需要加强的约束。
- 本轮必须保留的内容。
- 本轮修改范围。

### 3. 重生成保护机制

重生成时默认应保护：

- 已锁定角色外观。
- 已锁定场景。
- 已锁定镜头首帧。
- 已确认对白文本。
- 已确认角色声音。
- 已成功且无需重做的镜头结果。

### 4. 重生成结果说明

系统应尽量让用户看懂：

- 这次改了什么。
- 沿用了什么。
- 为什么这次结果会更接近用户要求。

## 十四、锁定与版本机制

### 1. 锁定机制

用户确认后的关键结果必须可锁定，至少包括：

- 概要。
- 角色设定。
- 角色三视图。
- 场景参考。
- 镜头首帧。
- 对白脚本。
- 角色声音。

当前已落地或已接近落地的锁定点包括：

- 长篇概要 / 章节概要。
- `character_turnaround`。
- `shot_first_frame`。
- `audio_script.audio_script_locked`。
- `media_assets.meta.locked` 下的对白、旁白和其他素材。

### 2. 版本机制

所有关键生成物应保留版本，至少包括：

- 小说概要版本。
- 正文版本。
- 分镜版本。
- 对白脚本版本。
- 音频版本。
- 视频片段版本。
- 最终视频版本。

起步阶段版本体系不必过重，但至少要支持：

- 查看历史结果。
- 恢复明确旧版本。
- 知道当前使用的是哪个版本。

## 十五、工程架构设计

### 1. 分层结构

建议系统分为以下五层：

#### 内容层

负责小说概要、正文、定稿等文字内容。

#### 约束层

负责项目级风格、角色、场景、镜头、音频和禁止项约束。

#### 生成层

负责分镜、对白脚本、配音、视觉素材、视频片段和最终合成。

#### 任务层

负责排队、执行、重试、取消、恢复和状态记录。

#### 反馈层

负责收集用户意见，并将其转为下一轮可执行约束。

### 2. 异步任务

以下任务应采用异步执行：

- 批量正文生成。
- 分镜生成。
- 自动对白脚本生成。
- 批量对白配音。
- 角色视觉素材生成。
- 视频片段生成。
- 最终视频合成。

### 3. 可观察性

每个任务都应至少能看到：

- 当前状态。
- 当前步骤。
- 已完成数量。
- 失败原因。
- 输出结果。
- 最近一次心跳。

### 4. 文件与素材管理

系统必须清楚管理以下资产：

- 角色视觉素材。
- 场景素材。
- 镜头首帧。
- 视频片段。
- 对白音频。
- 旁白音频。
- 字幕文件。
- 配乐。
- 音效。
- 最终视频。

后续应逐步从本地目录过渡到统一存储服务。

## 十六、当前能力与现实判断

结合当前实现情况，系统已经具备一条最小闭环：

- 小说概要生成与修订。
- 正文生成与定稿。
- 分镜生成与编辑。
- 自动对白脚本生成。
- 项目级视觉风格锁定。
- 角色三视图生成。
- 镜头首帧生成、预览和锁定。
- 角色声音配置字段。
- 对白音频生成与试听。
- 视频任务创建与执行。
- 视频片段生成。
- 对白混入镜头片段。
- 字幕文件生成。
- 最终视频合成。

并且当前实现已经新增几个关键的“可控性”能力：

- 视频任务、分镜任务、批量正文任务都开始输出结构化真实进度摘要。
- 本地视频分支会直接复用已锁定首帧。
- 即梦视频分支会在有首帧时优先走图生视频-首帧，否则回退文生视频。
- 预检流程会返回这次“生成了什么 / 跳过了什么”的真实结果，而不是只返回完成状态。
- 镜头编辑保存时不再意外清空 `character_refs / scene_refs`。

这说明系统已经从“概念设计”走到“最小可运行链路”。

但当前更重要的问题是产品化问题，而不是继续扩展零散功能。

## 十七、当前关键缺口

当前关键缺口主要在视频侧：

- 角色一致性还不够稳。
- 场景一致性还不够稳。
- 镜头控制还不够细。
- 字幕、配乐、音效尚未完整进入最终成片。
- 首帧已进入链路，但更完整的图生视频首尾帧、运镜参数和更强首帧约束还未接齐。
- 视频任务失败后的单镜头恢复能力还不够完整。
- 用户虽然能看到部分状态，但还不够直观。
- 结果虽可生成，但离“持续逼近用户要求”还有距离。

另外还有几个直接影响质量的现实问题：

- `scene_refs` 目前前端仍是最小可用编辑形态，还不是资产选择器。
- 即梦图生视频当前先接首帧模式，首尾帧和运镜模式尚未纳入统一 provider 分支。
- 角色、场景、首帧三类约束虽然已经开始叠加，但还没有形成完全结构化的镜头约束界面。

## 十八、近期建设顺序

### 第一阶段：真实有声样片跑通

目标：

- 用真实项目、真实章节跑通一个可预览的有声样片。

重点：

- 保证主链路稳定。
- 保证对白真正进入最终视频。
- 保证失败原因可见。

### 第二阶段：单镜头可控与失败恢复

目标：

- 把失败影响范围缩小到单镜头。
- 支持单镜头重试。
- 支持从失败点继续。

### 第三阶段：角色与首帧控制增强

目标：

- 强化角色一致性。
- 引入首帧确认。
- 降低镜头漂移。

这一阶段现在已经部分进入实现，不再只是规划：

- `character_turnaround` 已可生成与锁定。
- `shot_first_frame` 已可生成、预览、锁定。
- 本地视频分支已开始真实复用首帧。
- 即梦视频分支已开始优先使用首帧走图生视频-首帧。

### 第四阶段：成片包装增强

目标：

- 接入字幕烧录。
- 接入配乐与音效。
- 增强最终成片质量。

## 十九、产品完成判断标准

当以下条件基本满足时，系统才可以认为从“工程样机”进入“可持续使用产品”：

- 用户可以顺畅完成从项目设定到最终视频的主流程。
- 小说生成可以支撑后续视频生产，不成为主瓶颈。
- 视频生成能够稳定体现用户指定的主要方向。
- 用户可以对关键结果进行局部修改和重生成。
- 锁定内容不会被后续任务误覆盖。
- 用户可以单独重试失败镜头。
- 用户可以预览和下载最终有声视频。
- 用户不需要依赖后台日志才能理解系统状态。

## 二十、结论

本项目后续的核心，不是把小说和视频两侧都做成同样重的系统，而是明确：

- 小说侧以够用、可修订、可支撑视频为目标。
- 视频侧以可控、一致、可反馈、可重生成为目标。

系统真正要解决的问题，不是“能不能生成”，而是：

- 能不能越来越接近用户想要的效果。
- 能不能在复杂视频约束下仍然保持简单使用。
- 能不能让用户的意见真正改变下一轮结果。

因此，本项目的正式方向可以概括为三点：

1. 小说轻约束，视频重约束。
2. 默认简单，按需增强。
3. 以反馈、锁定和重生成作为核心能力建设方向。
