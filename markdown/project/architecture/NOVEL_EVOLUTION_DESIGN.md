# 晨流写作台：小说“演化机制”设计文档

> 历史说明
>
> 本文保留了早期“演化数据写回 GraphRAG 检索层”的设计讨论。
> 当前工作台实现仍保留演化抽取与正典更新，但不再以 GraphRAG 作为当前业务主链路的运行前置。

## 文档目的

这份文档只回答一件事：

`小说中的人物、关系、事件和世界认知，应该如何随着章节生成持续演化？`

这里的“演化”不是泛泛而谈，而是：

- 每章写完后，系统怎么识别变化
- 变化应该存成什么结构
- 变化如何进入 GraphRAG / 检索层
- 下一章生成时，如何优先利用这些变化

这份文档是后续落代码时的设计基准。

---

## 一句话定义

演化机制可以定义为：

`每生成一章 -> 提取本章变化 -> 写回状态层 -> 下一章生成时优先读取这些变化`

如果没有“写回状态层”，小说里的人物就永远只是初始设定的复制品。

---

## 为什么必须做演化机制

长篇小说真正好看的地方，不是静态设定，而是变化：

- 某个人做了一个糟糕的决定
- 某段关系从亲近转向疏离
- 某个群体开始不信任主角
- 某个误会留下了长期后果
- 某个角色表面没变，但内在状态已经变了

如果这些变化没有被系统化保存，下一章就很容易出现：

- 角色像失忆了一样
- 冲突没有后果
- 关系推进不连续
- 舆论和世界反馈消失

所以“演化”不是附加高级功能，而是长篇小说成立的基础设施。

---

## 演化不是“重写整个人设”

这一点必须先定清楚。

错误做法：

- 每生成一章就重写整个人物设定
- 每次都重新改写整份世界观
- 每章都更新所有角色的完整状态

这样做的代价是：

- 流程很重
- 状态容易相互冲突
- 很难控制历史一致性

正确做法：

`只更新“发生了变化的部分”`

也就是说：

- 不重写整个人
- 不重写整个世界
- 只记录这一章新增了哪些变化

---

## 演化对象应该拆成哪几层

当前建议拆成 4 层。

## 1. 核心设定层

这层很少变。

它描述：

- 角色是谁
- 世界基本规则是什么
- 长期不能违背的设定是什么

### 典型内容

人物：

- 基本性格底色
- 身份与背景
- 说话风格
- 核心价值观
- 长期创伤 / 执念

世界：

- 世界规则
- 学校 / 城市 / 组织的基础设定
- 时间背景

### 特征

- 更新频率低
- 更接近“长期设定”
- 不应该每章重写

---

## 2. 动态状态层

这层是演化机制的核心。

它描述：

- 角色现在正处于什么状态

### 典型内容

- 当前情绪
- 当前目标
- 当前立场
- 当前对某人的态度
- 当前对某件事的判断
- 当前在回避什么 / 执着什么

### 例子

不是：

- “他是一个温柔的人”

而是：

- “他现在因为内疚而刻意显得冷淡”
- “他表面嘴硬，但已经开始怀疑自己的决定”

### 特征

- 每章都可能变化
- 不替代核心设定层
- 是下一章最应该优先检索的内容

---

## 3. 事件层

这层描述：

- 到底发生了什么

### 典型内容

- 谁做了什么决定
- 谁对谁说了什么关键话
- 哪场冲突发生了
- 哪个秘密暴露了
- 哪个物件出现 / 回收了

### 事件层的意义

事件层不是为了“记录流水账”，而是为了提供：

- 因果链
- 连续性依据
- 关系变化的来源

### 特征

- 每章新增
- 是动态状态和外界认知变化的原因

---

## 4. 外界认知层

这层描述：

- 别人 / 某个群体现在怎么看这个人

这层非常重要，因为人物演化不只是“自己怎么变”，还包括“别人如何重新理解他”。

### 典型内容

- 女主对男主的信任下降
- 班级开始疏远主角
- 某组织更警惕他
- 朋友对他的判断从“可靠”变成“危险”

### 特征

- 不需要更新“整个世界”
- 只更新关键群体和关键角色的认知变化

---

## 推荐的数据模型思路

现有模型里已经有：

- `Project`
- `Memory`
- `SourceDocument`
- `GenerationRun`
- `Novel`
- `NovelChapter`

但还没有专门承载“演化”的数据层。

建议新增 4 类结构。

## 1. CharacterStateSnapshot

用于记录角色某一阶段的动态状态摘要。

建议字段：

- `id`
- `project_id`
- `character_key`
- `chapter_ref`
- `emotion_state`
- `current_goal`
- `current_stance`
- `self_view_shift`
- `summary`
- `created_at`

用途：

- 检索“这个角色最近变成什么样了”

## 2. RelationshipStateUpdate

用于记录关系变化。

建议字段：

- `id`
- `project_id`
- `source_key`
- `target_key`
- `chapter_ref`
- `change_type`
- `direction`
- `intensity`
- `summary`
- `created_at`

用途：

- 检索“这两个人最近的关系变化”

## 3. StoryEvent

用于记录重要事件。

建议字段：

- `id`
- `project_id`
- `chapter_ref`
- `title`
- `summary`
- `participants_json`
- `location_key`
- `impact_summary`
- `created_at`

用途：

- 建立章节因果链
- 给 GraphRAG 提供事件节点

## 4. WorldPerceptionUpdate

用于记录群体认知变化。

建议字段：

- `id`
- `project_id`
- `subject_key`
- `observer_group_key`
- `chapter_ref`
- `change_summary`
- `direction`
- `created_at`

用途：

- 记录“班级 / 家人 / 组织 / 城市舆论”的变化

---

## 最小版本不要一开始全做

为了避免系统过重，建议按层次上线。

### 第一阶段

只做：

- `CharacterStateSnapshot`
- `RelationshipStateUpdate`

这已经能显著提升小说连续性。

### 第二阶段

再做：

- `StoryEvent`

这样角色变化就有因果可追踪。

### 第三阶段

最后再做：

- `WorldPerceptionUpdate`

这样“世界怎么看角色”的变化才会进入系统。

---

## 每章后应该怎么提取变化

这一层建议做成专门的“章节后分析器”。

不是让 writer 顺手写，而是单独一个 extractor / updater prompt。

它的任务不是写文，而是输出结构化 JSON。

## 输入

给它的输入应包括：

- 当前章节正文
- 当前章节摘要
- 本章写作目标
- 当前角色与关系上下文
- 最近几章状态变化摘要

## 输出

建议输出结构：

```json
{
  "character_updates": [
    {
      "character": "男主",
      "emotion_state": "自责但嘴硬",
      "current_goal": "避免和女主正面冲突",
      "self_view_shift": "开始怀疑自己是否真的做对了",
      "summary": "表面强撑，内里已经松动。"
    }
  ],
  "relationship_updates": [
    {
      "source": "女主",
      "target": "男主",
      "change_type": "trust_shift",
      "direction": "down",
      "intensity": 2,
      "summary": "信任下降，开始有意识地拉开距离。"
    }
  ],
  "event_updates": [
    {
      "title": "男主在会议中牺牲同伴利益",
      "summary": "他为了保住更大的目标，做出了让同伴失望的决定。",
      "impact_summary": "关系出现明确裂痕。"
    }
  ],
  "world_updates": [
    {
      "subject": "男主",
      "observer_group": "班级舆论",
      "direction": "negative",
      "change_summary": "更多人开始觉得他不可靠。"
    }
  ]
}
```

---

## 为什么要做结构化提取

因为只有结构化，后续才能：

- 写回数据库
- 生成检索文本
- 进入 GraphRAG 索引
- 精准控制“哪些信息该被下章优先使用”

如果只是让模型说一段自然语言总结，后面还是会重新变成一坨难检索文本。

---

## 演化如何写回 GraphRAG

GraphRAG 不需要直接“理解角色成长”，它只需要索引正确的演化文本。

建议做法：

## 1. 把变化写成独立输入文档

例如每章生成后，额外生成：

- `state_chapter_0003.txt`
- `event_chapter_0003.txt`
- `relationship_chapter_0003.txt`

内容像这样：

```text
角色状态更新
角色：男主
章节：3
当前情绪：自责但嘴硬
当前目标：避免与女主正面冲突
状态变化摘要：他开始意识到自己的决定是有代价的。
```

```text
关系变化更新
来源角色：女主
目标角色：男主
章节：3
变化：信任下降
摘要：她开始刻意与他保持距离。
```

## 2. 下次重建 GraphRAG 输入时一并纳入

当前系统已经会重建：

- project profile
- memory
- source

后面应再纳入：

- recent state updates
- relationship updates
- event summaries

这样 GraphRAG 在索引时就能看到“最新演化结果”。

---

## 下一章检索时，应该优先检索什么

“演化”真正起作用，关键不在于记录，而在于下一章如何使用。

建议下一章生成时，检索优先级这样排：

## 第一优先级：最近 1 到 3 章状态变化

优先看：

- 当前角色最近变成什么样
- 当前关系最近怎么变

## 第二优先级：最近关键事件

优先看：

- 刚发生过什么
- 这些事带来了什么后果

## 第三优先级：核心设定层

优先看：

- 不可违背的人设与世界规则

也就是说：

`最近变化 > 最近事件 > 长期设定 > 大范围参考资料`

如果顺序反过来，小说就会重新变回“静态设定驱动”。

---

## 推荐的写作上下文卡结构

为了让演化真正被 writer 用起来，GraphRAG 查询结果不应该直接原样塞给模型。

建议增加一个“写作上下文卡”整理层。

结构建议如下：

```text
当前场景写作卡

一、当前目标
- 这一场戏要推进什么

二、关键角色当前状态
- 男主：自责但嘴硬，正在回避冲突
- 女主：失望，刻意保持距离

三、当前关系张力
- 女主对男主信任下降
- 两人还没有正面谈开

四、最近关键事件
- 第 3 章里男主做出了伤害同伴利益的决定

五、不能违背的连续性事实
- 女主已经知道这件事
- 男主还没有正式道歉

六、可用物件 / 场景线索
- 便利店门口
- 那把借过一次的透明伞
```

这比直接给 writer 一堆 local/global 原始文本更适合小说生成。

---

## 演化机制为什么不会无限变重

这是实施时最常见的顾虑。

要避免系统过重，建议加三个限制。

## 限制 1：每章最多提取 3 到 8 条关键变化

不要让 extractor 每章写几十条。

## 限制 2：只更新变化，不重写全量

例如：

- 更新“当前状态”
- 不重写“基础性格全文”

## 限制 3：检索优先只看最近几章

不要每次把全历史全部塞给模型。

建议：

- 默认重点看最近 3 章
- 再补核心设定层

---

## 当前项目的最小落地顺序

如果现在要开始做，不建议一次做完所有演化层。

推荐顺序如下：

## 第 1 步

先把“长期记忆”改成“长期设定”。

这一步是产品理解层的修正。

## 第 2 步

新增章节后分析器，输出：

- `character_updates`
- `relationship_updates`

先不做世界舆论变化。

## 第 3 步

把这些更新写回数据库，并生成增量 GraphRAG 输入文本。

## 第 4 步

生成时新增“写作上下文卡”整理层。

先不要直接让 writer 吃原始 local/global 文本。

## 第 5 步

再扩展：

- 事件层
- 群体认知层

---

## 最终结论

“演化”这件事，正确实现方式不是：

- 让模型自己记住
- 或者每次重写整个人设

而是：

1. 把角色 / 关系 / 事件 / 外界认知拆层
2. 每章之后只提取关键变化
3. 把变化写回状态层和 GraphRAG 输入
4. 下一章优先使用这些变化

真正的核心不是“角色设定变了”，而是：

`角色状态变了，关系温度变了，事件后果留下了，外界看法也变了。`

这才是小说里的演化。

如果后续真的开始实现，建议先从：

- `长期设定`
- `角色状态更新`
- `关系变化更新`

这三件事开始。

这是成本最低、收益最大的第一阶段。
